## 1. 平台概述与测试人员定位

### 1.1 平台能力概览

MeterSphere 是新一代开源持续测试平台。本仓库为**米多内部自研版**，测试人员日常使用的核心模块如下：

| 模块 | 前端路径 | 测试人员主要用途 |
|------|---------|----------------|
| 工作台 | `/workstation/*` | 查看待办、关注项、快捷入口 |
| 用例管理 | `/case-management/*` | 编写/维护功能用例、参与评审 |
| 测试计划 | `/test-plan/*` | 创建计划、执行用例、生成报告 |
| 接口测试 | `/api-test/*` | 接口调试、场景自动化、Mock |
| 缺陷管理 | `/bug-management/*` | 提交/跟踪/验证缺陷 |
| 项目设置 | `/project-management/*` | 环境、成员、模板配置 |
| Agent 集成 | `/setting/system/agentIntegration` | Token 申请（系统管理员操作） |

### 1.2 测试人员角色分工

| 角色 | 核心职责 | 本 SOP 重点章节 |
|------|---------|----------------|
| **测试工程师** | 用例编写、手工/Agent 执行、提 Bug、回归 | §4–§6、§8 |
| **测试负责人** | 测试策略、计划制定、质量门禁、Token 审批 | §5、§9 |
| **自动化工程师** | MCP 配置、Playwright 脚本、Agent 环境维护 | §8 |

### 1.3 数据分层（必须理解）

```
系统（System）
  └── 组织（Organization）  ← 业务隔离，切换组织后项目列表会变
        └── 项目（Project）  ← 日常测试工作上下文
              ├── 功能用例
              ├── 测试计划
              ├── 接口测试
              └── 缺陷
```

**操作前必确认**：左上角已选中正确的**组织**和**项目**，否则会出现「列表为空」「权限不足」等假象。

## 2. 环境与账号准备

### 2.1 环境检查清单

新成员接入或迭代开始前，确认以下项：

```
□ 前端可访问（开发环境 http://localhost:5173，测试/生产以实际域名为准）
□ 后端可访问（默认 http://localhost:8081）
□ 账号已创建并加入目标组织、项目
□ 项目角色含：用例读写、计划执行、缺陷读写
□ 测试环境 URL / 账号已在「项目设置 → 环境管理」配置
□ 前端社区版限制已解除（VITE_MS_UNLIMITED=true，自研版默认已配置）
```

### 2.2 测试账号与权限

| 场景 | 建议角色 | 最低权限 |
|------|---------|---------|
| 日常功能测试 | 测试成员 | `FUNCTIONAL_CASE:READ+UPDATE`、`PROJECT_TEST_PLAN:READ+UPDATE`、`PROJECT_BUG:READ+UPDATE` |
| 用例设计 | 测试成员 | `FUNCTIONAL_CASE:READ+ADD+UPDATE` |
| 测试负责人 | 项目管理员 | 项目全部权限 |
| Agent 协作 | 测试工程师个人 Token | `FUNCTIONAL_READ` + `FUNCTIONAL_SUBMIT` 或 `FUNCTIONAL_ALL` |

### 2.3 测试环境配置

**路径**：项目设置 → 环境管理

| 配置项 | 规范 |
|--------|------|
| 环境名称 | `{产品}-{环境}`，如 `工单系统-测试` |
| Base URL | 被测系统前端/后端入口 |
| 变量 | 公共账号、Cookie、Header 等 |
| 说明 | 用例「前置条件」中引用环境名，便于 Agent 和人工对齐 |

## 4. 功能用例管理 SOP

> **执行角色**：测试工程师（模块规划由测试负责人 + PM 共同确认）

### 4.1 模块规划规范

模块结构直接影响**人工检索效率**和 **Agent 自然语言命中率**，必须规范维护。

| 规范 | 正确示例 | 错误示例 |
|------|---------|---------|
| 按业务域划分顶层 | `登录/` `订单/` `支付/` | `测试1/` `杂项/` |
| 层级不超过 3 层 | `订单/创建/正常流程` | `订单/PC/Chrome/V1/...` |
| 命名简洁无歧义 | `用户管理` | `用户相关的一些功能` |
| 与产品功能模块对齐 | 参照 PRD 目录 | 按个人习惯随意命名 |

**Agent 模块别名（P2）**：若团队常用缩写（如「CW」指财务），由测试负责人协调系统管理员在 `agent_module_alias` 或对应管理 API 中配置，提升 Agent 检索命中率。

### 4.2 用例编写 SOP

#### 4.2.1 创建流程

```
Step 1：进入 用例管理 → 功能用例
Step 2：选择目标模块（必须挂模块，否则 Agent 无法模块检索）
Step 3：新建用例，填写：
  ├─ 用例名称：动词 + 对象 + 场景，如「创建订单-正常流程」
  ├─ 前置条件：环境、数据、权限要求
  ├─ 步骤模式：优先使用 Step 模式（操作 + 预期一一对应）
  ├─ 优先级：functional_priority 字段 P0/P1/P2/P3
  └─ 标签：smoke、regression、P0 等统一规范
Step 4：关联需求/Story（如有）
Step 5：保存并提交评审（迭代用例）
```

#### 4.2.2 用例质量检查清单

| 检查项 | 要求 |
|--------|------|
| 可执行性 | 每步操作明确，预期结果可验证 |
| 独立性 | 用例间无强耦合，可单独执行 |
| 数据说明 | 前置条件写清测试账号、初始数据 |
| 优先级 | P0 = 核心流程/冒烟；P1 = 主要功能；P2/P3 = 次要/边界 |
| 标签 | 至少打 `functional_priority` 对应标签或字段 |
| 模块归属 | 必须挂在正确模块下 |

#### 4.2.3 Text 模式用例说明

若用例为 Text 模式（非 Step），Agent 检索时会转为**虚拟步骤**并附带 warning。  
**建议**：所有 Agent 可执行用例统一改为 Step 模式，便于逐步执行与回写。

### 4.3 用例评审配合 SOP

测试工程师职责：

```
收到评审通知
  → 检查被分配用例的覆盖度与步骤完整性
  → 根据 PM/开发意见修改被驳回用例
  → 评审全部通过后，将用例纳入测试计划
```

评审入口：**用例管理 → 用例评审**（路径 `/case-management/caseManagementReview`，勿使用已废弃的 `caseReview` 路由）。

## 5. 测试计划与执行 SOP

### 5.1 测试计划创建 SOP

> **执行角色**：测试负责人创建，测试工程师配合关联用例

#### 5.1.1 人工迭代计划

| 步骤 | 操作 | 命名规范 |
|:--:|------|---------|
| 1 | 测试计划 → 创建计划 | `{迭代}-{类型}-{日期}`，如 `Sprint12-功能测试-20260713` |
| 2 | 关联用例 | 按模块/标签批量关联本迭代范围 |
| 3 | 分配执行人 | 可按模块或优先级分配 |
| 4 | 设置起止时间 | 与迭代节奏对齐 |
| 5 | PM 确认范围 | 确保 P0 需求均有对应用例 |

#### 5.1.2 Agent 专用测试计划（必建）

| 规范 | 说明 |
|------|------|
| 独立计划 | 名称如 `Agent-功能测试-2026`，与人工计划分开 |
| 预先关联 | 所有 Agent 可执行用例**提前**关联到此计划 |
| 不重复关联 | 同一计划内一条用例只关联一次 |
| 固定 planId | 配置到 MCP 环境变量 `MS_TEST_PLAN_ID`，Agent 检索自动带入 |
| 定期维护 | 新增 Agent 可执行用例时，同步关联到该计划 |

> **关键**：Agent 回写依赖 `testPlanCaseId`（计划关联 ID）。用例未关联 Agent 专用计划时，检索结果无 `testPlanCaseId`，无法写入计划执行历史。

### 5.2 手工执行 SOP

```
Step 1：进入测试计划详情 → 功能用例 Tab
Step 2：筛选待执行 / 指派给自己的用例
Step 3：逐条打开用例，按步骤执行：
  ├─ 每步标记：通过 / 失败 / 阻塞
  ├─ 失败时填写实际结果
  └─ 必要时上传截图
Step 4：失败用例 → 关联已有缺陷 或 新建缺陷
Step 5：全部执行完毕 → 查看计划进度与通过率
Step 6：通知测试负责人生成报告
```

### 5.3 执行结果判定规范

| 结果 | 适用场景 |
|------|---------|
| **通过（SUCCESS）** | 实际结果与预期一致 |
| **失败（ERROR）** | 实际结果与预期不符，缺陷可复现 |
| **阻塞（BLOCKED）** | 环境/数据/依赖问题导致无法继续，非产品缺陷 |
| **跳过** | 用例不适用当前版本（需注明原因） |

**注意**：阻塞不等于通过，不可为了通过率而标记通过。

### 5.4 测试报告 SOP

> **执行角色**：测试负责人生成，测试工程师配合确认数据

```
测试计划执行完毕
  → 测试计划 → 报告 → 生成报告
    → 选择关联计划、配置报告内容（通过率、缺陷分布等）
      → 生成报告链接
        → 提交 PM 审阅验收
```

报告生成前检查：

- [ ] P0 用例已全部执行
- [ ] 失败用例均已关联缺陷或注明阻塞原因
- [ ] Agent 执行的关键用例已人工抽查
- [ ] 遗留风险（P2/P3 未关闭缺陷）已记录

## 6. 缺陷管理 SOP

### 6.1 缺陷生命周期

```
新建（测试）→ 确认（开发）→ 修复中（开发）→ 已修复（开发）→ 验证（测试）→ 关闭
                                    ↓
                               拒绝/延期（需 PM 仲裁）
```

### 6.2 提交缺陷 SOP

```
Step 1：缺陷管理 → 创建缺陷
Step 2：填写必填项：
  ├─ 标题：【模块】现象简述
  ├─ 严重程度：P0/P1/P2/P3
  ├─ 复现步骤：可独立复现的操作序列
  ├─ 期望 / 实际结果
  ├─ 环境：测试环境 URL、浏览器版本等
  └─ 附件：截图、日志、录屏
Step 3：关联用例（从测试计划执行页关联，或在缺陷中关联用例）
Step 4：指派开发负责人
Step 5：P0/P1 缺陷即时通知测试负责人和开发负责人
```

### 6.3 缺陷严重级别标准

| 级别 | 定义 | 测试人员处理 |
|------|------|-------------|
| **P0-致命** | 系统崩溃、数据丢失、核心流程不可用 | 立即升级，阻塞发版 |
| **P1-严重** | 主要功能异常，无可靠绕行 | 当前迭代必须修复并回归 |
| **P2-一般** | 次要功能异常，有绕行方案 | 记录并排期 |
| **P3-轻微** | UI/文案/体验问题 | 记录待优化 |

### 6.4 缺陷验证与回归 SOP

```
收到「已修复」通知
  → 在原始环境按复现步骤验证
    → 通过：关闭缺陷，执行关联用例回归
    → 未通过：重新打开，附验证说明和新截图
      → P0/P1 回归需覆盖关联模块冒烟用例
```

### 6.5 缺陷记录模板（对外/跨团队）

```markdown
### BUG-XXX 【P级】标题
- **模块**：
- **类型**：功能 / UI / 接口
- **复现步骤**：
- **期望结果**：
- **实际结果**：
- **关联 API**：`METHOD /path`（接口类缺陷）
- **关联用例**：
- **环境**：
```

## 7. 接口测试 SOP

### 7.1 适用场景

| 场景 | 推荐入口 |
|------|---------|
| 单接口快速验证 | 接口测试 → 接口调试 |
| 接口文档化管理 | 接口测试 → 接口定义 |
| 多接口串联自动化 | 接口测试 → 接口场景 |
| 前后端分离联调 | Mock 服务 |

### 7.2 接口场景维护 SOP

```
Step 1：在「接口定义」中维护或导入 API
Step 2：在「接口场景」中编排调用顺序、断言、变量提取
Step 3：关联测试环境（Base URL、认证 Header）
Step 4：本地调试通过后保存场景
Step 5：纳入测试计划或 CI 流水线（如有）
Step 6：迭代变更时同步更新场景与断言
```

### 7.3 接口测试执行规范

| 检查项 | 要求 |
|--------|------|
| 断言 | 至少断言 HTTP 状态码 + 关键业务字段 |
| 环境隔离 | 禁止在生产环境执行写操作场景 |
| 数据清理 | 造数场景需配套清理步骤或说明 |
| 报告 | 执行后查看场景报告，失败项分析日志 |

## 8. AI Agent 协作测试 SOP

> Agent 详细技术说明见：[SOP-MeterSphere-Agent协作平台操作规范.md](./SOP-MeterSphere-Agent协作平台操作规范.md)  
> Cursor 接入步骤见：[cursor-onboarding.md](../task/metersphere_agent/cursor-onboarding.md)

### 8.1 Agent 协作定位

| 人工测试 | Agent 协作 | 推荐分工 |
|---------|-----------|---------|
| 探索性测试、新功能首测 | 已稳定用例的回归 | 新用例人工首测，稳定后纳入 Agent 计划 |
| 复杂业务 judgment | 批量 P0 冒烟 | Agent 每日冒烟，人工抽查 |
| 缺陷确认 | 固定流程回归 | Agent 结果需人工复核关键用例 |

**原则**：Agent 执行结果 ≠ 最终质量结论，仅作为效率辅助。

### 8.2 测试人员首次接入 Agent（Cursor + MCP）

#### 8.2.1 前置条件

- [ ] MeterSphere 后端可访问
- [ ] 已向测试负责人/系统管理员申请 Agent Token
- [ ] Agent 专用测试计划已创建，目标用例已关联
- [ ] 本机 Node.js >= 18
- [ ] 已安装 Cursor IDE

#### 8.2.2 申请 Token

**方式一（推荐，P2 UI）**：系统管理员在 **系统设置 → Agent 集成 → Token 管理** 创建。

**方式二（Legacy SQL）**：由系统管理员执行 SQL 插入 `agent_token` 表。

申请信息模板：

| 字段 | 示例 |
|------|------|
| Token 名称 | `Cursor-张三-功能测试` |
| 关联用户 | 你的 MS 账号 |
| 默认项目 | 目标项目 ID |
| 权限 | `FUNCTIONAL_READ,FUNCTIONAL_SUBMIT` 或 `FUNCTIONAL_ALL` |

> Token 明文（`msat_xxx`）**仅创建时展示一次**，立即保存到密码管理器，禁止提交到 Git。

#### 8.2.3 配置 MCP

在 MeterSphere 仓库根目录复制 `.cursor/mcp.json.example` 为 `.cursor/mcp.json`：

```json
{
  "mcpServers": {
    "metersphere": {
      "command": "npx",
      "args": ["-y", "@midoo/metersphere-mcp"],
      "env": {
        "MS_BASE_URL": "http://localhost:8081",
        "MS_AGENT_TOKEN": "msat_你的Token",
        "MS_PROJECT_ID": "你的项目ID",
        "MS_TEST_PLAN_ID": "Agent-功能测试-2026的计划ID"
      }
    }
  }
}
```

配置完成后**重启 Cursor**，在 Settings → MCP 确认 `metersphere` 为绿色已连接。

#### 8.2.4 连接验证（3 步）

| 步骤 | 在 Cursor 中说 | 期望结果 |
|:--:|--------------|---------|
| 1 | 「列出当前项目有哪些功能用例模块」 | 返回模块树 |
| 2 | 「提取登录模块 P0 用例，包含完整步骤」 | 返回用例列表，含 `steps` 和 `testPlanCaseId` |
| 3 | （可选）「将用例 #xxx 回写为 SUCCESS」 | 平台测试计划执行历史可见 |

### 8.3 Agent 日常测试流程（标准 5 步）

```
Step 1：表述测试需求
  ├─ 明确：「提取订单模块 P0 未执行用例，执行后回写」
  └─ 模糊：「财务相关用例」→ Agent 应先 list_modules 请你确认

Step 2：确认检索范围
  ├─ 检查 Agent 返回的 matchedBy / matchedModules / total
  ├─ 关注 warnings（如 MODULE_NOT_MATCHED_KEYWORD_FALLBACK）
  ├─ total > 20 时，先让 Agent 摘要确认再拉详情
  └─ 确认每条用例含 testPlanCaseId

Step 3：外部执行
  ├─ Agent 在 MeterSphere 外按 steps 逐步执行（Playwright / 手工配合）
  ├─ 逐步记录 actualResult
  └─ 失败时保留截图（P2 支持 attachments 回写）

Step 4：回写结果
  ├─ Agent 调用 submit_functional_result（或 batch 批量回写）
  ├─ testPlanCaseId 必须用 search 返回值，不是 caseId
  └─ steps 中保留 search 返回的 step.id

Step 5：人工复核
  ├─ 在测试计划查看 Agent 执行历史
  ├─ 抽查失败用例，确认非环境问题
  └─ 失败用例按 §6 提交缺陷
```

### 8.4 常用 Agent 指令速查

| 你说 | Agent 行为 |
|------|-----------|
| 「有哪些业务模块」 | `list_modules` |
| 「提取 XX 模块 P0 用例」 | `search_functional_cases({ query: "XX", filters: { priority: ["P0"] } })` |
| 「找 P0 未执行的用例」 | `search(filters: { priority: ["P0"], lastExecuteResult: ["PENDING"] })` |
| 「只看摘要不要步骤」 | `search(includeSteps: false)` |
| 「执行完回写 MeterSphere」 | 外部执行 → `submit_functional_result` × N |
| 「查看用例 1001 详情」 | `get_functional_case({ caseId: "..." })` |

### 8.5 Agent 执行质量保障

| 阶段 | 测试人员动作 |
|------|-------------|
| 每日冒烟 | Agent 跑 P0 → 人工 spot check 2–3 条 |
| 迭代回归 | Agent 全量 → 人工复核所有失败项 |
| 上线前 | 关键 P0 用例必须人工再执行一遍 |
| 每周 | 抽查 Agent 执行记录，评估步骤覆盖与结果准确性 |

### 8.6 关键字段对照（回写必知）

| 字段 | 说明 | 常见错误 |
|------|------|---------|
| `caseId` | 功能用例在用例库中的 ID | 与 testPlanCaseId 混淆 |
| `testPlanCaseId` | 用例在测试计划中的关联 ID | 回写时错填 caseId 导致平台无记录 |
| `steps[].id` | 步骤 ID | 回写时丢失导致步骤结果无法对应 |
| `steps[].expected` | 预期结果 | 来自平台 `result` 字段 |
| `lastExecResult` | 用例整体结果 | 任一步 ERROR/BLOCKED 则整体 ERROR |

## 10. 异常处理与故障排查

### 10.1 平台常见问题

| 现象 | 原因 | 处理 |
|------|------|------|
| 项目/用例列表为空 | 组织或项目上下文错误 | 重新选择组织 → 进入项目 |
| 用例评审 404 | 使用了废弃路由 | 使用 `/case-management/caseManagementReview` |
| 计划执行历史为空 | 用例未真正执行或未关联计划 | 确认执行操作完成；Agent 场景检查 testPlanCaseId |
| 工作台无数据 | orgId 缺失 | 先选组织再选项目 |
| 无法创建多组织 | 社区版限制未解除 | 联系管理员检查 `VITE_MS_UNLIMITED` |

### 10.2 Agent / MCP 常见问题

| 现象 | HTTP 码 | 处理 |
|------|:--:|------|
| MCP 未连接 | — | 检查 `mcp.json` 路径、Node 版本、重启 Cursor |
| 认证失败 | 401 | Token 无效/过期/被禁用，重新申请 |
| 回写权限不足 | 403 | Token 需 `FUNCTIONAL_SUBMIT` 或 `FUNCTIONAL_ALL` |
| 缺 testPlanCaseId | 200 | 用例未关联 Agent 专用计划；配置 `MS_TEST_PLAN_ID` 并关联 |
| 检索无结果 | 200（total=0） | `list_modules` 确认模块名；用 `filters.moduleIds` 精确检索 |
| 模块降级 warning | 200 | 模块名未精确命中，已降级 keyword；确认 matchedModules |
| 回写成功但平台无记录 | — | 检查是否错用 caseId 代替 testPlanCaseId |

### 10.3 升级路径

```
测试人员发现问题
  → 判断 P0–P3
    → 产品缺陷：提 Bug，按 §6 跟踪
    → 平台/Agent 问题：查 §10.1/§10.2
      → 未解决：联系测试负责人
        → 仍未解决：记录到缺陷清单 / 提交 issue（附请求体、响应、截图）
```

## 11. 术语表

| 术语 | 说明 |
|------|------|
| **功能用例** | 手工测试用例，含步骤与预期结果 |
| **Step 模式** | 逐步操作 + 预期，Agent 可直接按步执行 |
| **Text 模式** | 文本描述型用例，Agent 会转为虚拟步骤 |
| **测试计划** | 一次测试活动的用例集合与执行调度 |
| **testPlanCaseId** | 用例在测试计划中的关联 ID，Agent 回写必填 |
| **caseId** | 功能用例在用例库中的 ID |
| **Agent Token** | Agent 接入凭据，格式 `msat_xxx` |
| **MCP** | Model Context Protocol，Cursor 与 MeterSphere 的通信协议 |
| **functional_priority** | 用例优先级自定义字段（P0/P1/P2/P3） |
| **smoke** | 冒烟标签，最高优先级快速验证 |
| **regression** | 回归标签，发版前全量或增量验证 |
